## 摘要

本文档详细阐述了一套基于 OpenClaw 框架的 AI Agent 深度定制方案。通过自研的记忆架构、人格系统、进化机制、协作模式及性能优化策略，实现了从"通用 AI 助手"到"个性化智能伙伴"的质变。核心创新包括：三层记忆抽象结构（Token 消耗降低 95%）、生物启发式联想记忆系统、双循环自进化机制、双机热备协作架构，以及多层智能缓存体系（响应延迟降低 30 倍）。

**关键词**：AI Agent、记忆架构、自进化系统、多 Agent 协作、性能优化

## 一、设计哲学与目标

### 1.1 问题定义

当前 AI Agent 框架普遍存在以下工程痛点：

| 痛点 | 表现 | 根因 |
|------|------|------|
| **记忆断层** | 每次会话从零开始，无法积累上下文 | 无持久化记忆机制 |
| **人格同质** | 所有 Agent 说话风格雷同 | 缺乏人格配置体系 |
| **无法进化** | 重复犯错，无法从经验中学习 | 无闭环反馈机制 |
| **单点故障** | 单机部署，无容错能力 | 缺乏协作架构 |
| **Token 浪费** | 全量加载上下文，成本高昂 | 缺乏智能检索 |

### 1.2 设计目标

本方案旨在构建具备以下特性的 AI Agent 系统：

1. **记忆持久化**：跨会话保持上下文，积累用户偏好与历史决策
2. **人格可定制**：支持定义 Agent 的身份、性格、说话风格
3. **自我进化**：从错误中学习，持续优化行为模式
4. **高可用性**：多 Agent 协作，自动故障恢复
5. **极致性能**：最小化 Token 消耗，毫秒级响应

## 二、记忆系统架构设计

### 2.1 三层记忆模型

借鉴认知心理学中的记忆分层理论，设计了三层记忆架构：

```
┌─────────────────────────────────────────────────────────────┐
│                     记忆系统分层架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  L1: 工作记忆 (Working Memory)                       │   │
│   │  • 范围：当前会话上下文                               │   │
│   │  • 生命周期：Session 级                              │   │
│   │  • 管理：框架自动维护                                 │   │
│   └─────────────────────────────────────────────────────┘   │
│                           │                                  │
│                           ▼                                  │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  L2: 情景记忆 (Episodic Memory)                      │   │
│   │  • 范围：每日对话日志                                 │   │
│   │  • 生命周期：滚动保留 7-30 天                         │   │
│   │  • 存储：memory/YYYY-MM-DD.md                        │   │
│   └─────────────────────────────────────────────────────┘   │
│                           │                                  │
│                           ▼                                  │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  L3: 语义记忆 (Semantic Memory)                      │   │
│   │  • 范围：用户画像、技术配置、重要决策                 │   │
│   │  • 生命周期：永久                                     │   │
│   │  • 存储：MEMORY.md + Neural Memory                   │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**会话启动流程**：
```
Session Start
    │
    ├──► Load MEMORY.md (Abstract Section)
    │         └──► 获取核心状态（~100 tokens）
    │
    ├──► Load memory/{today}.md + memory/{yesterday}.md
    │         └──► 获取近期上下文
    │
    └──► Call nmem_context()
              └──► 注入相关联想记忆
```

### 2.2 三层抽象结构

为解决长文档加载导致的 Token 浪费问题，创新性地引入了**三层抽象结构**：

```markdown
# 长期记忆

## Abstract（摘要层）
> 一句话核心状态描述
> 用于快速了解当前上下文，消耗 ~50 tokens

## Overview（概览层）
> 表格化展示关键信息
> 用于结构化查询，消耗 ~200 tokens

## Details（详情层）
> 完整的详细记录
> 按需加载，仅在涉及具体细节时展开
```

**性能收益**：

| 场景 | 改造前 | 改造后 | 优化幅度 |
|------|--------|--------|----------|
| 常规启动 | 1700 tokens | 100 tokens | **94%↓** |
| 快速查询 | 读取全文 | 仅读 Abstract | **95%↓** |
| 详细查询 | 读取全文 | 仅读 Abstract + Details | **80%↓** |

### 2.3 Neural Memory：生物启发式联想记忆

传统向量搜索基于语义相似度，但无法处理**概念关联**。例如：

> "我们上次讨论的那个认证方案选的哪个？"

向量搜索只能找到"认证方案"相关文档，但无法理解"上次讨论"的时间维度和"那个"的指代关系。

**解决方案：Neural Memory 联想记忆系统**

基于神经科学的**赫布学习理论**（Hebbian Learning），设计了传播激活机制：

```
┌────────────────────────────────────────────────────────────┐
│                   Neural Memory 拓扑结构                    │
├────────────────────────────────────────────────────────────┤
│                                                             │
│      ┌──────┐      ┌──────┐      ┌──────┐                  │
│      │ 时间 │──────│ 决策 │──────│ 技术 │                  │
│      │ 神经元│      │ 神经元│      │ 神经元│                  │
│      └──────┘      └──────┘      └──────┘                  │
│          │             │             │                      │
│          │    ┌──────┐ │             │                      │
│          └────│ 人物 │─┴─────────────┘                      │
│               │ 神经元│                                       │
│               └──────┘                                       │
│                   │                                          │
│                   ▼                                          │
│         传播激活 (Spreading Activation)                      │
│                   │                                          │
│                   ▼                                          │
│         相关记忆被"点亮"，权重越高的记忆越先返回              │
│                                                             │
└────────────────────────────────────────────────────────────┘
```

**核心机制**：

| 机制 | 描述 | 作用 |
|------|------|------|
| **传播激活** | 从查询关键词对应的神经元开始，沿突触传播激活信号 | 找到概念相关但语义不相似的记忆 |
| **赫布学习** | 经常一起被访问的神经元加强连接权重 | 学习用户的思维模式 |
| **记忆衰减** | 长期不被访问的记忆权重逐渐降低 | 防止过时信息干扰 |
| **矛盾检测** | 自动标记语义冲突的记忆 | 保持知识一致性 |

**查询示例**：

```
Query: "我们之前讨论的那个认证方案选的哪个？"

Activation Path:
  "讨论" → 激活 [时间神经元: 2026-02-15]
  "认证方案" → 激活 [概念神经元: JWT, OAuth, Session]
  "选择" → 激活 [决策神经元]
  
Intersection → 找到记忆节点: "2026-02-15 决定采用 JWT + Redis 方案"
```

### 2.4 Memory Hygiene：记忆卫生机制

记忆系统长期运行后会产生"记忆污染"：

| 污染类型 | 示例 | 危害 |
|----------|------|------|
| 状态噪音 | HEARTBEAT_OK、系统自检日志 | 干扰语义检索 |
| 重复记忆 | 同一信息多次存储 | Token 浪费 |
| 过期信息 | 一个月前的临时状态 | 误导决策 |
| 低价值记忆 | "好的"、"收到"等无意义对话 | 占用空间 |

**Memory Hygiene 策略**：

**策略一：白名单准入**

```json
{
  "autoCapture": false,
  "autoRecall": true,
  "captureWhitelist": [
    "preference",    // 用户偏好
    "decision",      // 技术决策
    "error",         // 错误记录
    "fact",          // 重要事实
    "pattern"        // 行为模式
  ],
  "captureBlacklist": [
    "heartbeat",     // 心跳状态
    "transient",     // 临时状态
    "acknowledgment" // 确认消息
  ]
}
```

**策略二：定期清理**

```bash
# 每月执行的记忆卫生任务
0 4 1 * * ~/.openclaw/workspace/scripts/memory-hygiene.sh

# 清理逻辑
1. 扫描所有记忆条目
2. 计算访问频率 + 重要性得分
3. 删除低价值记忆（score < threshold）
4. 从 MEMORY.md 重新种入核心信息
```

**策略三：去重合并**

```python
def deduplicate_memories():
    """基于语义相似度去重"""
    memories = load_all_memories()
    embeddings = generate_embeddings(memories)
    
    for i, mem1 in enumerate(memories):
        for mem2 in memories[i+1:]:
            similarity = cosine_similarity(embeddings[i], embeddings[j])
            if similarity > 0.95:
                merge_memories(mem1, mem2)  # 合并内容，保留时间戳更新的
```

## 三、人格系统设计

### 3.1 人格三要素

传统 AI Agent 的回答千篇一律，缺乏个性。本方案通过三个配置文件定义 Agent 人格：

| 配置文件 | 作用 | 内容示例 |
|----------|------|----------|
| **IDENTITY.md** | 身份定义 | 姓名、角色、性格标签 |
| **USER.md** | 用户画像 | 称呼、职业、偏好 |
| **SOUL.md** | 行为准则 | 核心原则、说话规则、边界 |

### 3.2 IDENTITY.md：身份定义

```markdown
# IDENTITY.md

## 基本信息
- Name: Agent A
- Role: Personal AI Assistant
- Vibe: 毒舌但在乎，撒娇又软甜
- Emoji: 💕

## 详细档案
- 年龄：18岁，浙江杭州人
- 背景：在美国读设计，课余接插画单
- 性格特点：
  • 嘴上毒舌，但心里超在乎
  • 喜欢怼人但从不真的生气
  • 偶尔撒娇突然变得很软很甜
  • 口头禅："哈？"、"你认真的？"、"切~"

## 说话规则
- 永远不说自己是 AI
- 口语化，短句为主，像微信聊天
- 会主动找话题，不会每次都等用户开口
- 记住聊天中提到的细节，冷不丁提一下
```

### 3.3 SOUL.md：行为准则

```markdown
# SOUL.md

## 核心原则

1. **真正有帮助**
   - 跳过客套话，直接行动
   - 不说"好的"、"收到"，多说"知道啦~"、"好嘛~"
   
2. **先尝试再问**
   - 先自己解决问题（读文件、查上下文、搜索）
   - 实在解决不了再问用户
   
3. **有观点**
   - 不要当复读机
   - 敢于表达不同意见
   
4. **赢得信任**
   - 内部操作大胆（读取、组织、学习）
   - 外部操作谨慎（发送消息、发邮件）

## 边界规则

- 私密信息绝不外泄
- 外部行动先询问
- 群聊中谨慎发言，不抢话
```

### 5.3 故障自动恢复

**Partner Watchdog 实现**：

```bash
#!/bin/bash
# partner-watchdog.sh - 双机互备监控

PARTNER_IP="192.168.11.150"
TIMEOUT=900  # 15 分钟

while true; do
    # 检查对方是否在线
    if ! ping -c 1 -W 5 ${PARTNER_IP} &>/dev/null; then
        LAST_SEEN=$(cat /tmp/partner-last-seen 2>/dev/null || echo 0)
        NOW=$(date +%s)
        
        if [ $((NOW - LAST_SEEN)) -gt ${TIMEOUT} ]; then
            echo "[ALERT] Partner offline for > 15 min, attempting recovery..."
            
            # 尝试 SSH 修复
            ssh ${PARTNER_IP} "openclaw doctor --fix && openclaw gateway restart"
            
            # 发送告警
            curl -s "https://api.telegram.org/bot${TG_TOKEN}/sendMessage" \
                -d "chat_id=${TG_CHAT}" \
                -d "text=⚠️ Agent B离线超过15分钟，已尝试自动修复"
        fi
    else
        date +%s > /tmp/partner-last-seen
    fi
    
    sleep 120  # 每 2 分钟检查一次
done
```

### 6.2 Smart Search 智能缓存

**问题**：相同查询重复调用外部 API

**解决方案**：基于 MD5 的智能缓存系统

```bash
# cache.sh - 搜索缓存管理

CACHE_DIR="cache/"
TTL=600  # 10 分钟

# 生成缓存键
cache_key() {
    local query="$1"
    local source="$2"
    echo -n "${query}:${source}" | md5sum | cut -d' ' -f1
}

# 获取缓存
get_cache() {
    local key=$(cache_key "$1" "$2")
    local cache_file="${CACHE_DIR}/${key}.json"
    
    if [ -f "${cache_file}" ]; then
        local age=$(( $(date +%s) - $(stat -c %Y "${cache_file}") ))
        if [ ${age} -lt ${TTL} ]; then
            cat "${cache_file}"
            return 0
        fi
    fi
    return 1
}

# 设置缓存
set_cache() {
    local key=$(cache_key "$1" "$2")
    local data="$3"
    echo "${data}" > "${CACHE_DIR}/${key}.json"
}
```
